Case Study 01 — CSR 현장 운영
CSR · STRATEGY 분당서울대학교병원 / 용인세브란스병원 (2023–2025)
국가 자살예방사업 · 안전망사업 — 위험군 사전 발굴 워크플로 표준화
사건 발생 후 대응이 아니라, 사전에 위험군을 발굴해 사례 등록 목표 120%를 달성한 케이스.
📌 Context

분당서울대·용인세브란스에서 보건복지부·지자체 위탁 자살예방·안전망사업의 현장 실무를 담당했습니다. 위탁사업 특성상 분기·연간 사례 등록 KPI가 명확했고, 표면적인 사건 대응만으로는 사업 목표 달성이 어려운 상황이었습니다.

⭐ Problem Definition

"사건이 발생하면 대응한다"가 아니라, "전 입원·외래 환자 중 위험군을 어떻게 사전에 발굴해서 사례관리에 등록시킬 것인가"가 본질적 문제였습니다. 의료진의 임상 판단에만 의존하면 위험군이 누락되었고, 누락은 곧 사업 KPI 미달과 환자 안전 사각지대로 이어졌습니다.

⚖️ Key Considerations
Tradeoff Analysis

의료진 부담 vs 표준 스크리닝 — 모든 환자에게 PHQ-9·C-SSRS를 일괄 적용하면 의료진 업무 부담 증가. 입원 단계의 간호 평가에 자동 트리거되는 형태로 부담을 최소화

현장 즉시 대응 vs 외부 자원 연계 — 병원 내 단기 개입만으로는 퇴원 후 재시도를 막기 어려움. 정신건강복지센터·생명사랑위기대응 등 지역 자원 연계까지 워크플로에 포함

사례 수 채우기 vs 사후관리 품질 — 등록 건수를 늘리는 것이 목적이 아니므로 3·6·12개월 추적 모니터링까지 포함한 품질 지표를 같이 운영

🎯 My Role
  • 5단계 사례관리 워크플로 표준화 (스크리닝 → 트리아지 → 다직군 협업 → 외부 연계 → 사후관리) 현장 리드
  • 정신건강의학과·간호·복지 3자 사례회의 정례화 및 회의록 표준 양식 제작
  • 국고보조금 운영, 분기·연간 결과 보고서 작성, 표준 운영 매뉴얼 직접 작성
  • 분당서울대 모델을 용인세브란스 안전망사업으로 확장 이관 시 인계 자료 제작
🔧 Approach
  • 입원 시 간호평가에 PHQ-9·C-SSRS 자동 트리거를 결합해 위험군 누락을 구조적으로 차단
  • 위험도 3단계(고/중/저) 트리아지 기준을 다직군 합의로 명문화해, 사례 등록과 외부 연계 의사결정의 일관성 확보
  • 국고보조금 정산·결과보고서를 데이터 기반으로 작성, 위탁기관에 사업 성과를 정량적으로 증명
📎 검토 자료 (현장 운영 산출물 재구성)
5단계 사례관리 워크플로 — 스크리닝부터 사후관리까지 다직군 협업 표준화 및 위탁사업 KPI 달성 구조
Operating Note

"위험군 사전 발굴 +47%, 사례 등록 목표 120%."

분당서울대 안전망사업에서 정착시킨 워크플로를 용인세브란스로 이관. 의료진 업무 부담을 늘리지 않고 KPI를 달성한 핵심은 간호평가 자동 트리거와 3자 사례회의 정례화였습니다.

💡 Decision

의료진의 개별 판단에 의존하던 위험군 발굴을, 병원 운영 시스템에 내장된 표준 워크플로로 재정의했습니다. 입원 평가·트리아지·외부 연계·사후관리를 단일 사례관리 표로 통합하고, 의료진·간호·복지가 같은 기준으로 합의할 수 있는 구조를 만들었습니다.

Case Study 02 — B2B 제휴·정산 운영
EXECUTION · DATA 네이버클라우드 NAVER WORKS · CPS팀 (2025.06–)
앱마켓 파트너 6개사 월 정산 — 교차검증 프로세스 정착
파트너 신고 매출과 내부 집계 매출의 산식 불일치 문제를, 월 단위 교차검증 표준으로 풀어낸 케이스.
📌 Context

네이버클라우드 Cloud Product Synergy(CPS)팀에서 NAVER WORKS 앱마켓 파트너 6개사의 월 정산·대사 업무를 담당했습니다. 월 평균 억 단위 정산이 발생하고, 파트너별로 과금 단가·수수료 산식·이용량 측정 기준이 조금씩 달라 정산 분쟁이 발생할 수 있는 구조였습니다.

⭐ Problem Definition

"누가 잘못 계산했는가"가 아니라, "파트너 신고 매출 vs 내부 집계 매출이 다를 수 있는 모든 산식·이상치를 어떻게 한 표 위에서 동일하게 검증할 것인가"가 본질적 문제였습니다. 한 번의 분쟁은 정산 신뢰뿐 아니라 파트너십 자체에 영향을 주기 때문입니다.

⚖️ Key Considerations
Tradeoff Analysis

속도 vs 정합성 — 마감 기한을 맞추기 위해 파트너 신고치를 그대로 신뢰하면 빠르지만 분쟁 위험. 산식별 재계산을 마감 전에 끝내야 하므로 점검 항목 우선순위 명확화 필요

전수 검증 vs 표본 검증 — 모든 거래를 일일이 확인하면 인적 비용 증가. 이상치 룰 기반 자동 추출 + 임계치 초과 항목만 직접 파트너 확인하는 하이브리드 방식 채택

파트너 관계 vs 데이터 원칙 — 차이를 발견했을 때 "파트너 편의로 넘기기"는 단기 우호 관계 유지에 도움이 되지만, 누적되면 신뢰가 더 빠르게 무너지므로 원칙 우선

🎯 My Role
  • 월 정산 교차검증 체크리스트 표준화 — 파트너 신고/내부 집계/단가 재계산/세금·수수료 (실무 리드)
  • 이상치 자동 추출 룰 정의 후 SUMIF·VLOOKUP·피벗테이블로 검증 자동화
  • 서비스 종료 고객사의 End-to-End 안내·정산 처리 (메일 템플릿 정착)
  • 월 정산 보고서 작성 및 파트너 응대 (분쟁 대응 시 데이터 근거 제공)
🔧 Approach
  • "무엇이 일치해야 하는가"를 먼저 정의 — 신고 매출, 단가 × 사용량 재계산, 부가세·수수료 산식, 서비스 종료 분리, 이상치 4종 — 이를 단일 검증 시트로 통합
  • 표면적 산식 일치만이 아니라 산식의 출처(계약·가격표·운영 정책)까지 함께 기록해, 분쟁 시 즉시 근거를 제시할 수 있도록 구조화
  • 임계치를 초과하는 항목은 즉시 파트너와 직접 확인 — "오차 발견 → 24시간 내 파트너 회신" 운영 원칙 정착
📎 검토 자료 (운영 산출물 재구성)
정산 교차검증 대시보드 — 파트너 신고 매출 vs 내부 집계 매출 정합성 시각화 및 6단계 검증 체크리스트
Operating Note

"12개월 누적 — 정산 분쟁 0건."

단발성 검증이 아니라 매월 동일한 시트·동일한 체크리스트로 운영했고, 데이터 근거가 명확했기 때문에 가능했습니다. 해피빈의 기업 기부금 · 굿액션 정산에도 같은 운영 체계를 그대로 적용할 수 있습니다.

💡 Decision

"분쟁이 일어나면 대응한다"가 아니라, 분쟁 자체가 발생하지 않도록 매월 동일한 검증 시트를 운영하는 방향을 선택했습니다. 파트너 신고치 → 내부 재계산 → 산식 출처 기록 → 이상치 직접 확인의 4단 구조를 표준화했고, 12개월 누적 정산 분쟁 0건을 유지했습니다.

Case Study 03 — 서비스 기획·운영
PRODUCT · OPERATIONS 네이버클라우드 Cloud Product Synergy · 차세대 업무협업 플랫폼 (2025–)
NAVER WORKS 헬프센터·지원시스템 기획 — 한일 이중언어 운영 체계
CS 빈출 문의를 줄이려면 "FAQ를 더 만드는 것"이 아니라, 릴리즈노트–헬프센터–공지사항을 하나의 운영 표준으로 묶어야 한다.
📌 Context

NAVER WORKS는 한국·일본 두 시장을 동시에 운영하는 B2B SaaS입니다. 신규 기능 배포 시 릴리즈노트·헬프센터 문서·공지사항이 동시에 갱신되어야 하지만, 채널별로 작성 주체와 톤이 달라 사용자 입장에서 정보가 분절되고 CS 1차 응대 부담이 커지는 구조였습니다.

⭐ Problem Definition

"문서를 더 만든다"가 아니라, "릴리즈노트–헬프센터–공지사항이 같은 메시지·같은 한일 번역 기준으로 동시에 나가는 운영 체계를 어떻게 만들 것인가"가 본질적 문제였습니다. 운영 체계가 없으면 어떤 콘텐츠를 추가해도 다음 배포에 다시 같은 균열이 발생합니다.

⚖️ Key Considerations
Tradeoff Analysis

채널별 자율성 vs 운영 표준 — 채널마다 작성자가 다르고 톤이 달라야 하지만, 핵심 메시지·용어는 표준화되어야 사용자 혼란이 사라짐. 표준 템플릿 + 채널별 가이드로 분리

한국어 우선 vs 한일 동시 발행 — 한국어 먼저 발행 후 일본어 번역 운영은 시차 발생. "기능 배포 = 한일 동시 공개"를 운영 원칙으로 잡고 번역·검수 흐름 재설계

FAQ 추가 vs 셀프서비스 카드 설계 — 단순 FAQ 누적은 검색 정확도 저하. CS 빈출 주제를 카드형 셀프서비스로 재구성해 검색·해결 동선 단축

🎯 My Role
  • 릴리즈노트–헬프센터–공지사항 표준 템플릿 설계 (콘텐츠 구조·핵심 메시지·CTA 통일)
  • 한국어/일본어 동시 운영 가이드 작성 — 용어집 공유, 번역 검수 흐름 정의
  • CS 빈출 항목 분석 → 셀프서비스 카드 7종 신규 정착, 검색 동선 재설계
  • 릴리즈노트 작성 — 외부 사용자 관점에서 "변경된 가치"가 즉시 보이도록 표현 정리
🔧 Approach
  • 릴리즈노트가 변경 사실 나열로 끝나지 않도록 — "새로운 기능 / 개선사항 / 효과" 3분할 표준 도입
  • 한국어 본문이 확정되어야 일본어 번역이 시작되던 흐름을 "초안 동시 작성 → 동시 검수" 흐름으로 변경해 발행 시차 제거
  • CS 문의 데이터를 주제·빈도 기준으로 분석해, 검색 1회 만에 해결 가능한 셀프서비스 카드를 설계
📎 검토 자료 (운영 산출물 재구성)
릴리즈노트 표준 템플릿 — 새로운 기능 / 개선사항 / 효과 3분할 + 한국어·일본어 동시 발행 가이드
Operating Note

"문서가 아니라 운영 체계를 만든다."

해피빈에서도 마찬가지입니다. 굿액션 캠페인 페이지·기업 파트너 가이드·해피로그 후기가 따로 움직이면 사용자·기업 모두 메시지가 흐려집니다. 같은 운영 표준으로 묶을 수 있는 경험을 그대로 옮길 수 있습니다.

💡 Decision

FAQ 추가가 아니라, 릴리즈노트–헬프센터–공지사항을 하나의 운영 표준으로 묶는 방향을 선택했습니다. 표준 템플릿 + 한일 동시 발행 흐름 + 셀프서비스 카드 재설계의 3단 구조로 콘텐츠가 분절되지 않도록 운영 체계 자체를 바꿨습니다.

Case Study 04 — 신규 파트너십 제안
PARTNERSHIP · CSR 루닛케어 × 해피빈 · 자체 제안서 (2026.04)
헬스케어 AI × 해피빈 — 서비스 기부 모델 기반 신규 매칭
현금 기부가 아닌, 환자 1인 × AI 전문 상담 1회 = 의료 정보 격차 1건 해소. 해피빈에 비어 있는 "헬스케어 AI" 영역을 직접 채우는 5장 제안서.
📌 Context

지원 준비 과정에서 해피빈의 20년 공익 기록(해피로그)·해피 히어로즈 39편·기업 파트너십 사례를 직접 분석했습니다. 헬스케어 AI 기업 CSR이 거의 비어 있는 반면, 의료·AI 사용자 관심 댓글은 2,722건에 달한다는 점을 확인했습니다.

⭐ Problem Definition

"기업이 어떤 NGO에 얼마를 기부할 것인가"가 아니라, "기존 현금 기부로는 결합되지 않던 헬스케어 AI 기업의 서비스 자산과 해피빈의 검증된 NGO 생태계를 어떤 매칭 구조로 연결할 것인가"가 본질적 문제였습니다.

⚖️ Key Considerations
Tradeoff Analysis

현금 기부 vs 서비스 기부 — 현금은 회계·정산이 단순하지만 "기업의 강점"이 사용자 경험으로 전달되지 않음. 헬스케어 AI의 강점은 서비스이므로 서비스 기부 모델이 자연스러움

기업 직접 매칭 vs NGO 경유 배분 — 기업이 환자에게 직접 매칭하면 빠르지만 의료 정보 격차 환자 발굴 역량이 부족. 검증된 NGO를 경유해 정보 소외 계층 우선 배정 필요

본 사업 vs 파일럿 우선 — 헬스케어 CSR은 신규 영역이므로 12개월 본 사업화는 위험. 3개월 파일럿(이용권 500매)으로 KPI 검증 후 본 사업화 단계 설계

🎯 My Role
  • 해피빈 공개 데이터(해피로그·해피 히어로즈) 분석 및 시장 공백 식별 (전체 리드)
  • 3주체(해피빈·루닛케어·NGO) 매칭 구조 설계 — "굿액션 1건 = 이용권 1매" 단위 매칭
  • 3개월 파일럿(암 환자 세이프티 넷 캠페인) 시나리오 + 5단계 실행 STEP 설계
  • 3단계 12개월 로드맵(파일럿 → 확장 → 정착) + 4개 KPI(굿액션 참여율·이용권 사용율·NPS·해피로그 등록건) 정의
  • 5장 제안서 자체 디자인·작성 (라이트 모드 · 해피빈 그린 톤 · NanumSquareNeo)
🔧 Approach
  • 해피빈 해피 히어로즈 39편을 직접 읽고 의료·AI 관련 사용자 댓글 2,722건을 정성 분석해, 시장 공백을 정량적으로 증명하는 근거 자료로 활용
  • "환자 1인 × AI 전문 상담 1회 = 의료 정보 격차 1건 해소"라는 단위 매칭 공식을 만들어, 기존 현금 기부와의 차별점을 한 줄로 압축
  • 파일럿 KPI를 운영 가능한 4개로 좁히고, 각 단계(파일럿/확장/정착)별 산출물·기대 결과·후속 의사결정 기준을 명시
📎 자체 제작 제안서 (실제 산출물 · 5장)
01 · Cover — AI 시대, 달라지는 기업 CSR 파트너십 : 기록에서 연결로
02 · Opportunity — 20년 / 2,722건 / 공백 — 시장 공백 정량 증명
03 · Partnership Model — 해피빈 · 루닛케어 · 환자 3주체 매칭 구조
04 · Pilot — 3개월 파일럿 시나리오 + 5단계 실행 STEP
05 · Roadmap & KPI — 12개월 마스터 플랜 + 4개 KPI 정의
Author Note

제안서는 모두 직접 작성했습니다.

데이터 분석 · 매칭 구조 설계 · 슬라이드 디자인까지. 본 자료는 Claude Code + Google AI Studio를 활용해 작성했고, 입사 후에도 같은 AI 기반 업무 방식으로 파트너십 기획·성과 관리에 기여하고 싶습니다.

💡 Decision

현금 기부의 확장이 아니라, 서비스 기부 모델을 통한 새로운 매칭 영역 개척을 제안했습니다. 해피빈 20년 공익 기록(검증 자산) × 헬스케어 AI(서비스 자산) × 검증 NGO(전달 자산)의 3주체 매칭 구조를 정의했고, 3개월 파일럿 → 확장 → 정착의 단계별 로드맵으로 위험 부담을 분산했습니다.

Closing

CSR 파트너십을 하나의 사이클로 연결하고 싶습니다.

기업·NGO·수혜자를 데이터의 신뢰로 묶어내는 일.
의료 현장에서 5년, IT 운영에서 1년 — 두 세계를 직접 오가며 같은 운영 체계로 풀어 왔습니다.
해피빈에서 이 경험을 신규 파트너십 발굴 → 운영 → 성과 보고의 한 사이클로 연결하고 싶습니다.

"비록 한 기업의 CSR 사업을 처음부터 끝까지 단독 리딩한 경험은 아직 부족하지만,
현장 운영부터 정산·서비스 기획까지 각 단계의 실무 기반을 갖추었다고 자신합니다."

현장의 진정성을 운영 가능한 언어로

의료사회복지 현장에서는 "표면 사건 대응"이 아닌 "위험군 사전 발굴 워크플로"로, 루닛케어 제안서에서는 "현금 기부"가 아닌 "환자 1인 × AI 상담 1회 = 의료 정보 격차 1건 해소"로 — 진정성을 의사결정 가능한 단위로 다듬어 왔습니다.

억 단위 정산 무결성, 12개월 누적

네이버클라우드 앱마켓 6개사 정산을 매월 동일한 검증 시트로 운영해 분쟁 0건을 유지했습니다. 해피빈의 기업 기부금·굿액션 정산에 같은 운영 체계를 그대로 가져갈 수 있습니다.

다직군·다국가 합의 구조 운영

의료 현장에서는 정신건강의학과·간호·복지 3자 회의를, NAVER WORKS에서는 한·일 동시 발행 흐름을 운영했습니다. 해피빈의 기업·NGO·플랫폼 3자 합의 구조도 같은 방식으로 풀 수 있습니다.

신규 파트너십을 직접 설계하는 사람

지원 준비 과정에서 해피빈에 비어 있는 헬스케어 AI 영역을 직접 식별해 5장 제안서를 자체 제작했습니다. 입사 후에도 시장 공백 → 매칭 구조 → 파일럿 → 확장의 흐름을 직접 만들어내는 운영자가 되겠습니다.
